# VisionOps v3 整体架构

## 1. 架构目标

VisionOps v3 面向工业现场的视觉 AI 全生命周期，重点解决三个问题：模型如何稳定生产，模型如何可靠部署到 Rockchip 边缘设备，以及推理结果如何进入 Gateway、Modbus 和 PLC 业务链路。

架构采用控制面与数据面分离：Python 组件主要构成控制面，负责 Web、配置、训练、部署和诊断；C++ 边缘组件构成生产数据面，负责相机帧传输、RKNN 推理与实时结果交付。

## 2. 系统上下文

```text
                        服务端 / 训练环境
┌─────────────────────────────────────────────────────────┐
│ Server API / 数据管理 / 标注审核                         │
│                   │                                     │
│                   v                                     │
│ Training Pipeline -> ONNX -> RKNN -> 版本化模型包        │
│                   │                                     │
│                   v                                     │
│              Deployment Orchestrator                    │
└───────────────────┬─────────────────────────────────────┘
                    │ 模型包、配置、服务定义
                    v
                     RK3588 / LB3576 边缘设备
┌─────────────────────────────────────────────────────────┐
│ Camera Bridge -> C++ RKNN Runtime -> 标准化推理结果      │
│                         │                               │
│                         +-> Collector Web               │

│                         +-> Modbus Adapter -> PLC        │
│                                                         │
│ 状态、指标、日志和诊断信息 <- 各边缘组件                 │
└─────────────────────────────────────────────────────────┘
```

## 3. 分层架构

### 3.1 应用层

`apps/server_api/` 提供服务端 API、数据工作流和模型生命周期入口。`apps/collector_web/` 提供边缘设备本地 Web 管理界面，包括配置、状态、诊断、模型切换和结果展示。

应用层不实现 RKNN 推理，也不解析模型原始张量。

### 3.2 模型工程层

`training/pipeline/` 负责任务编排、数据预处理、训练、评估与实验记录。`training/export/` 负责 ONNX 导出、RKNN 转换、设备目标检查和发布清单生成。

输出不是散落的模型文件，而是带版本、任务类型、输入规范、类别、阈值建议、校验值和目标平台信息的模型包。

### 3.3 边缘运行层

`edge/camera_bridge/`、`edge/rknn_runtime/`、`edge/gateway_adapter/` 和 `edge/modbus_adapter/` 是独立部署单元。它们通过版本化协议通信，允许独立升级和故障隔离。

C++ RKNN Runtime 是生产推理核心。Collector Web 是管理者和观察者，不位于帧处理的硬实时路径中。

### 3.4 接口层

`interfaces/schemas/` 定义配置、模型清单、检测结果、健康状态和诊断信息的数据结构。`interfaces/protocols/` 定义进程间通信、HTTP、Unix Socket、共享内存或消息传输约定。

接口必须带版本。协议升级优先保持向后兼容；不兼容变更应提升主版本并提供迁移说明。

### 3.5 配置与部署层

`configs/` 保存可审查的源配置和运行时生成规则。`deploy/` 保存 systemd 单元、部署脚本和发布包定义。真实设备密钥、环境文件和模型不进入 Git。

## 4. 关键数据对象

- **FrameDescriptor**：帧标识、时间戳、尺寸、像素格式和传输句柄。
- **ModelManifest**：模型版本、任务、目标平台、输入输出规范和文件校验值。
- **InferenceResult**：帧标识、模型版本、预测结果、耗时和错误状态。
- **DeviceConfig**：设备身份、相机、运行服务、通信和资源限制。
- **HealthSnapshot**：组件状态、版本、依赖、队列深度和最近错误。

具体格式将在实现前写入 `interfaces/`，不得由单个模块私自定义同名结构。

## 5. 进程与故障边界

生产环境默认将 Camera Bridge、C++ RKNN Runtime、Collector Web 和通信适配器作为独立 systemd 服务运行。

- 相机断流不应导致 Web 服务退出。
- Web 重启不应中断已运行的推理链路。
- Gateway 或 PLC 不可用时，推理应继续运行，并按策略限量缓存或丢弃结果。
- 模型切换失败时应保留上一可用版本，禁止留下半更新状态。
- 每个组件提供独立健康检查、版本信息和结构化日志。

## 6. 平台适配

RK3588 与 LB3576 共用上层接口，但设备 SDK、NPU 核心策略、媒体管线、像素格式和性能参数可以不同。平台差异通过构建目标、设备能力描述和适配实现隔离。

业务代码只依赖抽象能力，例如“获取 NV12 帧”或“执行 RKNN 模型”，不直接判断设备型号。

## 7. 非目标

初始阶段不追求一次性覆盖 v2 的全部功能，不提供 Python 生产 RKNN 备选主链路，不兼容所有旧配置字段，也不将模型、数据或第三方 SDK 纳入仓库。
