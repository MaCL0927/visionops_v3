# M25.3 edge venv migration

VisionOps v3 previously inherited `/opt/visionops/venv` from v2. The affected
code was concentrated in edge launchers and production deployment templates.

## Legacy references found

- `scripts/start_collector.sh`
- `production/carton_line/scripts/start_{runtime,collector,gateway,ws_pick}.sh`
- `production/carton_line/deploy/install_services.sh`
- `production/carton_line/deploy/production.env.example`
- `production/carton_palletizing/scripts/start_*.sh`
- `production/carton_palletizing/deploy/production.env.example`
- existing `/etc/visionops_v3/carton_line.env` and
  `/etc/visionops_v3/carton_palletizing.env` generated by older installers
- `apps/collector_web/README.md`

The three carton-line watchdog scripts also invoked `python3` directly. They now
prefer the same v3 interpreter. Because they only use the standard library and
serve as recovery safeguards, they fall back to the board system Python if the
v3 venv itself is temporarily unavailable; they never fall back to the v2 venv.

## New default

```text
/opt/visionops_v3/venv/bin/python3
```

Launch scripts execute this interpreter explicitly and fail with an actionable
message if the environment has not been created. They no longer silently fall
back to the v2 venv or to an arbitrary system Python.

## Installation

After cloning the repository to `/opt/visionops_v3`:

```bash
cd /opt/visionops_v3
sudo bash scripts/setup_edge_env.sh
```

Optional development tools:

```bash
sudo bash scripts/setup_edge_env.sh --with-dev
```

Recreate an inconsistent environment:

```bash
sudo bash scripts/setup_edge_env.sh --recreate
```

The script installs general Ubuntu tools, creates a Python venv with access to
Ubuntu's aarch64 OpenCV/numpy packages, installs the remaining v3 Python
requirements, and validates Collector and production imports. It does not copy
repository code and does not install proprietary RKNN/RGA/camera SDKs.
