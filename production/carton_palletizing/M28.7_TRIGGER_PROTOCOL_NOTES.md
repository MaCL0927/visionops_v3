# M28.7 carton_palletizing trigger protocol integration

This revision adds trigger-mode robot communication to
`tasks/first_layer_placement` without changing the OBB model or the existing
`box_grasp_vision` / `tube_pick_vision` contracts.

Key points:

- WebSocket server: `0.0.0.0:9001/vision`.
- Video: existing 336L MJPEG stream, soft-synchronized with JSON.
- `pallet_place_target`: returns the current layer's next slot center, camera
  coordinate, and slot OBB angle.
- `held_box_pose`: returns the selected held-carton OBB center, camera
  coordinate, and detected long-axis angle.
- Initial phase: `held_box_selection.mode=nearest_depth`, normally with
  `layering.max_layers=2`.
- Later phase: `held_box_selection.mode=outside_tray`, with any desired layer
  count or `max_layers=0`.
- One external placement trigger may use multiple internal fresh frames to
  finish temporal occupancy confirmation and layer baseline capture, but only
  one detection message is returned to the robot.
- Odd layers use placement template A and even layers use the interleaved
  template B.  Layer geometry is generated from the stable tray footprint,
  rather than copying the previous layer's four OBB boxes unchanged.
- Periodic WebSocket `status` messages are configurable and disabled by
  default for this trigger-only task.  The mock robot client hides status
  unless `--show-status` is supplied.

See `tasks/first_layer_placement/PROTOCOL.md` for the complete contract.
